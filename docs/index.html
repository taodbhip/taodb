TaoDB

给 LLM agent 持久的记忆。一个二进制文件，零配置。

$ curl -fsSL https://raw.githubusercontent.com/taodbhip/taodb/main/install.sh | bash

然后 cd your-project && taodb init，重启 agent。完毕。

谁在用 TaoDB

💻 Coding

Agent 记住架构决策、bug 模式、模块历史。不用每次重新解释代码库。

📝 写作与文档

Agent 记住研究线索、草稿演变、章回大纲。长文创作的连贯性不丢失。

🎨 设计

Agent 记住设计系统规则、组件迭代、客户反馈。每次改版都有上下文。

🎬 视频与广告

Agent 记住项目 brief、剪辑决策、平台规格、客户修改意见。项目间生产线不断。

只要你在多个会话里用 LLM agent 做创作或技术工作，TaoDB 就是给你的。

LLM agent 是失忆的

每次会话从零开始。Claude Code 不记得上周修了什么 bug。Cursor Agent 不知道哪个架构决策导致了今天的代码结构。Windsurf 想不起三个会话前刚踩过的坑。

这不只是程序员的痛。写小说的人每次都要重新告诉 agent 主角叫什么、第几回发生了什么。设计师每次都要重新描述设计系统。做视频的人每次都要重新交代客户 brief。

向量数据库解决不了这个问题。它们给你"语义相似"的文本——但知道一段代码"关于认证"和知道"我们在 sprint 3 修了 token rotation 的竞态条件，这个修复引入了回归"是两回事。知道一段文本"关于主角"和知道"第 152 回主角发现鼓皮共鸣与矿脉有关"是两回事。

LLM 需要的是记忆。不是语义搜索。是时空记忆。

使用 TaoDB 前后的区别

没有 TaoDB

Agent: "我来看看 auth 模块..." Agent 从头读 auth.rs，不知道历史 Agent: "这里有个竞态条件，我修一下" Agent 不知道这个问题上周已修过 Agent: "用 mutex 锁一下" Agent 不知道上次用 mutex 引入了死锁

有 TaoDB

Agent: "taodb_recent(1)" → 上次：用 atomic swap 修复了 token rotation 竞态 Agent: "taodb_recall_constraints" → JWT 架构、AppError 约定、bcrypt 迁移决策 Agent: "看到了。已经有了上下文——继续。"

工作原理

不可见的本地 MCP 服务。无需端口、无需 token、无需配置。

taodb init → agent 通过 .mcp.json 发现 taoDB → agent 用 MCP stdio（标准输入输出）和 taoDB 通信 → taoDB 读写本地 taodb-memory/ 目录 → agent 有记忆了。整个过程对用户完全不可见。

三层记忆架构：

  • 约束层 — 规则、架构、设计系统、客户 brief。永不衰减。每次会话自动加载。
  • 叙事层 — 事件、变更、迭代、草稿。随时间衰减，近期优先。
  • 感官层 — 纹理、模式、情绪。跨容器、跨时间、跨项目触发联想。

重要内容设 energy_floor=0.7 永不消退。每次召回 +0.05 能量，使用强化记忆。

开始使用

$ curl -fsSL https://raw.githubusercontent.com/taodbhip/taodb/main/install.sh | bash

然后 cd your-project && taodb init，重启 agent。完毕。不需要启动任何服务器。

agent 通过本地 MCP stdio 自动连接 taoDB。零网络、零 token、零配置。

首次会话 agent 自动检测空数据库 → 提示导入项目内容 → 之后每次会话自动召回上下文。
