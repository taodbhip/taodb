# 核心概念

TaoDB 的记忆模型建立在认知科学基础上——而非向量相似度。理解这些概念，你才能为 agent 设计高效的记忆工作流。

## 核心洞察

LLM 擅长语义理解。LLM 不擅长持久化存储和时间排序。TaoDB 处理后者。LLM 处理前者。

这个分工是 TaoDB 最重要的设计决策：

```
LLM 做的事:    "这条记忆现在相关吗？"
              "这条记忆对当前任务意味着什么？"

TaoDB 做的事:  "第 143-148 回发生了什么？"
              "auth 模块最近 30 天有哪些记忆？"
              "世界规则永久保留，章回细节自然消退。"
```

## 四层记忆模型

每条记忆有 `potential_energy`（潜能，0.0-1.0）和 `energy_floor`（能量地板，它最低能降到多少）。

| 层级 | 能量范围 | 行为 |
|------|---------|------|
| **工作记忆** | 时间窗内全部可访问 | `recall(within_days=30)` 返回 |
| **长期记忆** | 衰减中，高于阈值可检索 | `recall(min_energy=0.3)` 返回高能记忆 |
| **永久记忆** | `energy_floor ≥ 0.5` — 永不低于地板 | 召回时自然命中；用于规则、架构、世界设定 |
| **遗忘记忆** | 低于阈值 | 不主动召回；痕迹保留，可被重新激活 |

## 能量地板

存储记忆时，你设置它的 `energy_floor`：

| 地板值 | 用途 | 示例 |
|--------|------|------|
| `0.0` | 普通内容——章回、代码改动、日常笔记。自然衰减。 | "把变量名从 x 改成 user_count" |
| `0.3` | 值得记住——转折点、修过的 bug、关键决策。 | "修了 token refresh 的竞态条件" |
| `0.5` | 半永久——角色档案、模块架构、团队规范。 | "Auth 模块采用 JWT + OAuth2 架构" |
| `0.7` | 永久——世界规则、设计原则、不可违背的约束。 | "绝不在环境变量里存密钥——用 vault" |

一条 `energy_floor = 0.7` 的记忆，无论过多久，能量永不低于 0.7。始终可通过 `recall(min_energy=0.5)` 找到。

## 时间索引

TaoDB 使用**叙事时间**，而非墙上时钟。这很重要。

**叙事时间**是你项目内部的时间线。对小说，是章回编号或故事内日期。对代码库，是 sprint 编号或版本号。绝对值不重要——只有相对顺序重要。

```bash
# 第143回: time_ns = BASE + (143 - 1) × 86400 × 10^9
# 第144回: time_ns = BASE + (144 - 1) × 86400 × 10^9
# 相对顺序决定召回时的排序
```

TaoDB 从容器标签自动推导 `time_ns`（识别 `第143回`、`sprint-5`、`day-30` 等模式）。大多数情况不需要你手动设置。

召回时，TaoDB 从你记忆库中最新的匹配记忆推导锚点时间——不用系统时钟。这意味着召回永远相对于你项目的当前位置。

## 空间索引（容器标签）

容器标签是按领域组织记忆的字符串标记。它们是你的空间坐标系统。

好的容器命名习惯：

```
编程:     module:auth, module:database, feature:oauth, sprint:2025-Q1
写作:     人物:桑安歌, 场景:邯郸酒肆, 物件:鼓, 第152回
知识管理:  topic:rust, topic:system-design, source:paper, source:meeting
```

引擎支持**模糊匹配**：如果你写 `module:Auth`，它自动修正为 `module:auth`（如果已存在）。这让标签空间保持一致，无需手动清理。

## 约束层 vs 叙事层

受 Shadow-Loom 论文启发，TaoDB 将记忆分为两层：

**约束层**（`energy_floor ≥ 0.5`，`memory_type = "constraint"`）：
- 世界规则、角色感知框架、架构决策
- 永不衰减
- 会话启动时通过 `taodb_recall_constraints` 自动注入 agent 上下文
- 例："Auth 模块使用 JWT，15 分钟轮换。禁止在 localStorage 中存储 token。"

**叙事层**（`energy_floor < 0.5`，`memory_type = "narrative"`）：
- 章回事件、代码改动、会议记录
- 随叙事时间衰减
- 通过 `taodb_recall()` 或 `taodb_recall_narrative()` 检索

这种分离确保永久知识永远可访问，而短暂事件自然消退——不会让旧代码改动和架构原则混在一起。

## 感官索引

受普鲁斯特"无意记忆"概念启发——一个感官触发可以激活跨越时间、空间、人物的记忆。

```bash
# 存储带感官锚点的记忆
taodb_memorize({
  "text": "鼓沿木缝在她指下发出涩感。干的，不规则的。",
  "senses": ["涩", "干"],
  "containers": ["人物:桑安歌", "物件:鼓", "第152回"]
})

# 之后——写另一个场景，出现类似的感官质地
taodb_recall_sensory(["涩", "干"])
# 返回所有共享这些感官的记忆——不论角色、章回、场景
```

感官索引是跨容器、跨人物、跨时间的。第 20 回的一个"涩"感可以激活第 152 回的"涩"记忆——实现那种让创作鲜活起来的意外联结。

## 重新巩固增强

每次记忆被召回，它的能量增加 0.05（上限 1.0）。这模拟了记忆重新巩固——使用记忆强化记忆。

实际效果：频繁使用的记忆保持可访问。被遗忘的记忆消退。系统自然浮现重要的内容。

## 衰减机制

采用 30 天（叙事时间）半衰期公式：

```
能量 = 强度 × 关联度 / (1 + 时间距离 / 半衰期)
```

其中时间距离是锚点与记忆之间的叙事时间差。30 叙事天前的记忆能量是当前记忆的一半。60 天前是三分之一。365 天后接近零——除非被 `energy_floor` 保护。

定期触发衰减：

```bash
taodb_decay  # 通过 MCP
```

被 `energy_floor` 保护的记忆不受影响。
