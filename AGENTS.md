# taodb — 设计哲学与架构

> *English summary: This file explains the design philosophy and architecture of taodb for contributors. See [README.md](README.md) for the user-facing project overview.*

## 核心定位

taodb 是 LLM agent 的长期记忆层。不是数据库。不是搜索引擎。不是向量存储。

- **LLM = 大脑**（决策、判断、创作）
- **Skills / System Prompt = 工作记忆**（当前任务的操作流程）
- **taodb = 长期记忆**（跨会话持久化的上下文）

只做 LLM 做不到的事：持久化存储 + 时空索引 + 记忆衰减。

## 设计原则

1. **不做语义理解。** LLM 拿回原始记忆后自己判断相关性。taodb 不替代 agent 的判断。
2. **不做 embedding。** 零外部 API 依赖、零向量存储、零 GPU 需求。一个二进制文件完成所有事。
3. **时空优先，语义为次。** "auth 模块上周发生了什么" 比 "找到跟这段文本语义相似的内容" 更有用。时间是 agent 工作流的一等线索。
4. **约束层与叙事层分离。** 受 Shadow-Loom 论文启发：世界规则/架构决策用 `energy_floor` 保护，事件/迭代随时间自然衰减。
5. **感官索引是对 embedding 的替代。** 受 Proust 无意记忆启发：感官触发比向量相似度更接近人类记忆的工作方式。

## 三层架构

### 约束层 (Constraint)
- 永不衰减。每次会话启动自动注入 agent 上下文。
- 存储：世界规则、架构决策、设计系统、项目约定、客户 brief。

### 叙事层 (Narrative)
- 随时间衰减（30 叙事日半衰）。
- 存储：事件、变更、决策、迭代、草稿。

### 感官层 (Sensory)
- 跨容器、跨时间、跨项目的触发维度。
- 存储：纹理、模式、情绪、触觉印象。

## 技术选型

| 组件 | 选择 | 理由 |
|------|------|------|
| 语言 | Rust | 零运行时、单二进制、内存安全 |
| 存储 | redb | 嵌入式 B-tree，零运维依赖 |
| 协议 | MCP stdio | agent 原生协议，零延迟、本地通信 |
| HTTP | axum | 非 MCP 集成场景的后备通道 |
| 序列化 | bincode + CRC32 | 紧凑二进制 + 完整性校验 |

## 关键论文

- **EM-LLM** — 贝叶斯惊异度做事件分割；两阶段检索（相似度+时间邻接）
- **Shadow-Loom** — fabula/syuzhet 双时间索引；约束化简报→渲染→审计
- **Proust (无意记忆)** — 感官触发比主动检索产生更强烈的体验复现
- **CoALA** — 模块化记忆：工作/情景/语义/程序

## 八卡体系（通鉴写作方法论）

taodb 的 MCP 工具设计遵循八卡方法论：

1. **写作核心卡** — 身体驱动叙事（POV 人物身体在场景中能感知到什么）
2. **框架构建卡** — 场景驱动架构（9±1 场景/卷）
3. **词汇卡** — 禁用现代词、商道用语、阶层称谓
4. **批量诊断卡** — 存量章回 QC（硬指标 + 叙事声音质量）
5. **落盘 QC 卡** — 读者视角三层检查（肉身在场/余味/意外）
6. **深度重写卡** — 记忆层/身体层/同时感知层三层深化
7. **CJK 补充卡** — 六类有机增补层（非灌水）
8. **50 回打分卡** — 十维度加权质量评估

每张卡的逻辑对应 taodb 的一个召回维度或记忆类型。
